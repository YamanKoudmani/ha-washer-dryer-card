# Flatten the Washer/Dryer Card Layout — Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Restyle the `washer-dryer-card` so the two machine panels have no per-section container chrome and content extends edge-to-edge horizontally, matching the flatter visual style of the older card reference. No JS, template, config, or version changes.

**Architecture:** Surgical CSS-only edits inside the single `static get styles()` block in `src/washer-dryer-card.ts`. Seven existing rules are modified and one new rule is added. The bundled output `dist/washer-dryer-card.js` is regenerated by `npm run build` and committed alongside the source change because Home Assistant loads the bundle, not the source.

**Tech Stack:** Lit 3, TypeScript, Rollup, Vitest. No new dependencies.

**Reference spec:** `docs/superpowers/specs/2026-06-14-flatten-washer-dryer-card-design.md`. Follow its §1–§9 exactly. Acceptance criteria are in the spec.

## Commit Strategy

The repo's commit history is small, focused, single-purpose commits (e.g. "Fix SVG namespace", "Bump version to 1.1.1"). This plan follows that style:

- **Commit A** — All CSS edits in `src/washer-dryer-card.ts` (Tasks 1–6 combined as one logical "restyle" commit).
- **Commit B** — Rebuilt `dist/washer-dryer-card.js` (Task 7).

If a reviewer prefers granular commits, the per-task commits are still possible — each task is independent. The plan defaults to the two-commit shape to match the spec's atomicity.

---

## Test Strategy

The project has no visual-regression tests for the card UI. The "test" for this change is a chain of three:

1. **`npx tsc --noEmit`** — confirms the TypeScript source still typechecks (the change is inside a `css` template literal, so this is mostly a sanity check).
2. **`npm run build`** — confirms Rollup produces a new bundle without errors.
3. **`npm test`** — confirms the existing `src/__tests__/utils.test.ts` still passes (it tests `utils.ts` only, but it must not regress).

Visual verification against the spec's acceptance criteria is **manual** and is the user's responsibility in their live Home Assistant instance (Task 9).

**Files:** none (manual user action)

- [ ] **Step 2: Open a dashboard that contains the washer-dryer card**

If the card is not yet on a dashboard, add it via the Lovelace UI ("Add Card" → search for "Washer Dryer Card" → use the existing config).

- [ ] **Step 3: Verify acceptance criteria 1–9 from the spec**

Walk through each acceptance criterion in `docs/superpowers/specs/2026-06-14-flatten-washer-dryer-card-design.md` (the "Acceptance Criteria" section). In particular:

- AC1: panels have no background, border, or rounded corners.
- AC2: `ha-card` outer container has zero padding on all four sides.
- AC3: a 1px hairline divider separates Washer from Dryer.
- AC4: the header has 14px top padding and aligns horizontally with panel content.
- AC5: hovering a panel gives a subtle row tint (or is disabled if unreadable).
- AC6: keyboard focus still shows the 2px primary-color focus ring (Tab to a panel).
- AC7: clicking a panel still opens the entity more-info dialog.
- AC8: the phase diagram SVG extends edge-to-edge inside its panel.
- AC9: the empty state (no machines configured) still looks correctly spaced — to test, temporarily hide both machines in the card config or open the editor on a card with no machines.

- [ ] **Step 4: Verify AC10 implicitly (the build already succeeded in Task 7)**

If the card loads at all in HA, AC10 is satisfied.

- [ ] **Step 5: If any acceptance criterion fails, report back**

If the row tint is imperceptible (AC5), the implementation plan's fallback is to remove the hover effect entirely:
```css

  background: none;
}
```
Document the deviation from the spec in the spec file as a follow-up note. Do not amend already-pushed commits unless the user explicitly asks.

If the divider color is invisible (AC3), try changing to:
```css

  border-top: 1px solid rgba(127, 127, 127, 0.3);

```
Then rebuild and re-commit per Task 7's pattern.

If the empty state looks cramped (AC9), check that `.empty-state` rule at lines 273–292 is unchanged and its `padding: 48px 16px` is intact. The plan does not modify that rule.
